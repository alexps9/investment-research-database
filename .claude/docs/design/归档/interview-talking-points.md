# 面试讲法：PaperGraph 项目

> 框架来源：功能层 → 系统层 → 工程层

---

## 第一层：功能层（30秒说清楚）

> "我做了一个学术论文技术路线可视化工具。输入一个技术方向关键词，系统自动抓取该领域引用量最高的论文，用引用关系构建知识图谱，识别技术社区，生成两种视图：时间轴视图看演进路径，聚类视图看论文亲疏。"

这一层讲完，面试官会问：**"你为什么做这个？"** —— 进入第二层。

---

## 第二层：系统层（重点，要能展开）

### 2.1 为什么做这个，痛点是什么

> 做投研/研究时，判断一个技术方向的演化脉络，现有工具要么是 Google Scholar（只能线性检索，没有关系视图），要么是 Connected Papers（只能以单篇论文为中心扩展，无法做方向级全景）。我想做的是：**输入一个方向，自动生成该方向的技术演化图**。

### 2.2 系统怎么拆模块

```
用户搜索 "transformer"
    ↓
[Concept 解析层] OpenAlex /concepts API → 关键词映射到学术概念 ID
    ↓
[论文抓取层] 按 concept + CS 领域过滤，cited_by_count 降序，取 top 100
    ↓
[图构建层] NetworkX 建有向图，只保留主论文之间的引用边
    ↓
[社区检测层] Louvain 算法，resolution=0.8，合并 singleton
    ↓
[命名层] TF-IDF 提取社区内论文标题关键词，生成社区语义名
    ↓
[可视化层] react-force-graph-2d，两种布局：fixed timeline / force-directed
```

### 2.3 关键设计决策（面试官最爱追问这里）

**为什么用 Concept ID 而不是关键词搜索？**

> 关键词搜索只匹配标题和摘要。"Attention Is All You Need" 标题里没有 "transformer" 这个词，用关键词搜就漏掉了。改用 OpenAlex 的 concept 标签之后，系统能识别出这篇论文属于 Transformer 概念，正确纳入结果。这是从"字面匹配"到"语义分类"的关键一步。

**为什么用 Louvain 而不是 K-means / 谱聚类？**

> 论文数量不固定，不知道应该分几个社区，所以不能用 K-means（需要预设 K）。Louvain 是图上的社区发现算法，直接作用于引用关系拓扑，不需要预设类别数，且时间复杂度 O(n log n)，100-1000 节点都能实时返回。

**时间轴视图为什么用 fixed position 而不是 d3 force？**

> 最初用自定义 d3 force（对每个节点施加年份方向的力），但实际测试中发现：100 个节点里有将近一半没有引用关系（isolated nodes），这些节点只受斥力没有引力，力模拟结束后会飘散到随机位置。最终改成直接给每个节点设置 `fx`/`fy`（fixed coordinates），完全绕过物理模拟，保证所有节点严格按年份列排列。

### 2.4 你怎么定义"可用"

> MVP1 的验收标准：搜 "transformer" 后，Swin Transformer、ViT、T5 这类引用量过万的核心论文必须出现在结果里；社区数量在 5-15 之间（太多说明图太稀疏，太少说明没有区分度）；图里边数 > 节点数（保证连接密度，否则可视化没意义）。

## 第三层：工程层（加分项）

**数据质量问题怎么处理？**
> OpenAlex 部分论文存在引用数虚高的数据错误（如一篇 2023 年数学论文显示 7 万引用，明显异常）。处理方式：建了黑名单 + 按发表年限做引用数合理性封顶（每年最多 10k）。这不完美，但对 demo 场景够用，真正的解法是接一个更可靠的论文数据库（如 Semantic Scholar）。

**性能**
> 搜索一次大约 3-5 秒，瓶颈在 OpenAlex 的网络 IO。后端用 asyncio + httpx 并发抓取（semaphore 限 10 并发），已经是单次请求的极限。下一步可以加 Redis 缓存热门搜索词。

**错误处理原则**
> 所有错误透明上浮，不 silent fail。API 超时、concept 解析失败、图构建失败，每一种都有独立的错误类和 actionable 的提示信息。

## 面试官追问预案

| 问题 | 回答要点 |
|------|---------|
| 为什么不用 Connected Papers？ | 它以单篇为中心，我想做方向级全景；且它不开放 API |
| Louvain 的 resolution 怎么调？ | 0.8 是实验值，太低社区太少（一个社区 60 篇），太高碎片化成 40+ 个 |
| 引用关系能代表技术相关性吗？ | 不完全能，这是 MVP1 的核心假设也是核心局限。MVP2 会加 SPECTER2 向量相似度来验证这个假设 |
| 为什么不直接用 embedding？ | 引用关系是明确的人类标注（作者主动引用 = 承认技术传承），比 embedding 余弦相似度有更强的因果语义；两者组合才是最优解 |

## MVP1 Demo 讲法（现场演示脚本）

**搜索 "transformer"，展示结果时说：**

> "这是该领域 2015-2024 年引用量最高的 100 篇论文。节点大小代表引用量，颜色代表技术社区——同一颜色是 Louvain 算法自动识别出的同一研究方向。橙圈是引用量 Top 10 的核心论文。"

**切换视图时说：**

> "时间轴视图能清晰看出演进路径，从左到右是 2015 到 2024 年。引用聚类视图更接近真实的知识结构，引用越密集的论文会被力引向同一个区域，可以看到论文之间的亲疏远近。"

**点击节点时说：**

> "点击任意节点可以看到论文详情，一键跳转 Semantic Scholar 查看原文。"

## 一句话收尾

> "这个项目的核心价值不是技术难度，而是把一个研究员做技术调研时脑子里的隐性认知——这些论文哪些有关系、哪个方向最近在爆——显式化、可视化地展示出来。MVP1 验证了引用关系作为信号的可行性，MVP2 用 SPECTER2 向量相似度验证语义距离是否和引用距离一致。"
