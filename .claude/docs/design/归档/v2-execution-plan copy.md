## 定位

HHNQ 体系核心板块。根据学术论文寻找技术路径的实现逻辑，构建"技术演进"可视化看板。

## 什么没变（继续沿用 thoughts.md 的核心思路）

这些是 V1 已经想清楚的，V2 直接继承：

1. **双层网络架构**：引用边(因果骨架) + 语义边(SPECTER2+文献耦合)
2. **向量化逻辑**：SPECTER2 语义嵌入 + 文献耦合强度，组合权重 α×相似度 + (1-α)×耦合
3. **社区检测**：Louvain/Leiden 自动聚类（V2 在此基础上叠加人工标签）
4. **MPA 主路径分析**：SPLC 算法提取知识流主干（V2 将其作为趋势预测的输入）
5. **技术增量标签**：LLM 对比相邻节点，回答"B 比 A 多做了什么"
6. **力导向可视化**：权重决定距离，因果决定方向

## V2 新增：精细化打标体系

### 为什么不能只靠 Louvain？
Louvain 是拓扑聚类，输出"社区 0/1/2"——没有语义。
新需求要求按**技术流派**归类，需要人工定义分类体系。

### 技术流派分类（初始版本）

| 流派 | 锚点论文 | 时间锚点 | 优先级 |
|-----|--------|---------|-------|
| Transformer | Attention Is All You Need | 2017 | P0 |
| BERT | BERT: Pre-training | 2018 | P0 |
| Mamba/SSM | Mamba: Linear-Time Sequence | 2023 | P0 |
| Vision Transformer | ViT: An Image is Worth 16x16 | 2020 | P1 |
| Diffusion | DDPM / Stable Diffusion 系列 | 2020+ | P1 |
| RWKV | RWKV: Reinventing RNNs | 2023 | P1 |
| MoE | Switch Transformers | 2021 | P2 |
| Flash Attention | Flash Attention 1/2 | 2022 | P2 |

### 标注策略（三级递进）
1. **关键词自动标注**：标题/摘要匹配规则（快，准确率 ~70%）
2. **LLM 辅助标注**：Claude 读摘要判断流派（中速，准确率 ~85%）
3. **人工验证**：秋恬 + PRD 团队抽样（慢，准确率 ~95%）

Phase A 用 1+3 验证逻辑，Phase B 起用 2+3 批量化。

### 标注数据结构
```json
{
  "paper_id": "W2741809807",
  "tech_tags": ["Transformer", "Attention"],
  "primary_tag": "Transformer",
  "confidence": 0.92,
  "tagged_by": "auto|llm|human"
}
```

## V2 新增：三件事报告

每个技术流派输出三个维度的洞察，这是 V2 的核心产出物。

### (1) 时空定位
> 该技术目前发展到了什么时间点？我们在坐标系的什么位置？

- 技术生命周期阶段（萌芽/成长/成熟/衰退）
- 论文发表时间分布（2013-2026 热度曲线）
- 引用增速（最近 12 个月加速度）
- 与其他流派的时间关系

**数据来源**：OpenAlex 的 publication_year + cited_by_count 时序数据
**实现**：后端聚合统计 → 前端时序图表

### (2) 人才图谱
> 最近在该方向研究的核心团队和人是谁？

- 核心作者排名（按论文数/引用数）
- 机构分布（大学/公司）
- 新进入者（最近 6 个月新加入的研究者）

**数据来源**：OpenAlex 的 authorships 字段（作者+机构信息免费可用）
**实现**：后端从论文数据提取作者-机构关系 → 聚合排名

### (3) 瓶颈破局分析
> 该阶段技术正在解决的具体问题是什么？下一步突破点在哪？

- 当前主要瓶颈（从论文标题/摘要提取高频问题词）
- 已有解决方案及其有效性（引用数/后续论文数）
- 未解决的问题 → 预测下一步

**数据来源**：论文标题/摘要的 TF-IDF + LLM 总结
**实现**：后端 NLP 提取 → LLM 生成摘要报告

## V2 新增：趋势预测

### 与 thoughts.md 的关系
thoughts.md 第三步"从拓扑图到技术演化路径"是**历史回顾**（MPA 提取已发生的路径）。
V2 在此基础上加一层**未来预测**。

### 预测方法
```
历史数据 (2013-2025)
  → 提取特征向量 (论文数/年, 引用增速, 作者数, 机构数)
  → 拟合增长曲线 (S 曲线 / 指数)
  → 预测 2026-2027 方向
  → 输出：热度预测 + 风险评估
```

### 预测指标
- 论文产出量趋势
- 引用增速趋势
- 技术融合预测（哪些流派在靠近）
- 风险评估（过热/衰退信号）

## MVP 定义：10 篇种子论文 × 4 条技术路径

### 为什么从这 10 篇开始

不用 OpenAlex 搜 100 篇再筛，而是**手动选定 10 篇关键论文**，原因：
1. 每篇论文在技术演化中的位置是确定的，不需要算法猜
2. 4 条路径的分叉关系清晰，直接验证"技术路线发现"的核心逻辑
3. 10 篇足够小，可以手工标注 + 手工验证三件事报告的质量
4. **技术含量不在"数据怎么拿"，而在"拿到后怎么分析和呈现"**

### MVP 验收标准

用这 10 篇论文，系统必须能输出：

#### 1. 可视化（已有能力的延伸）
- [ ] 力导向图中 10 个节点按 4 条路径聚类，颜色区分
- [ ] X 轴 = 年份（2023-2024），能看到时间演进
- [ ] 引用边方向正确（老 → 新）
- [ ] 路径 B 的 Jamba 悬浮在路径 A 和路径 B 之间（混合架构的视觉表达）

#### 2. 三件事报告（V2 核心新增）
对每条路径输出：

**路径 A 示例报告**：
```
时空定位：Attention 优化处于成熟期，2023-2024 从学术突破转向工业落地
         FlashAttention → Ring Attention → DeepSeek MLA，优化粒度越来越细
人才图谱：Tri Dao (Princeton→Together AI), Hao Liu (UC Berkeley), DeepSeek 团队
瓶颈分析：当前瓶颈是超长上下文的内存墙，Ring Attention 部分解决
         下一步可能方向：硬件-算法协同设计
```

- [ ] 4 条路径各有一份三件事报告
- [ ] 报告内容与论文实际贡献一致（人工验证）
- [ ] 报告能自动生成（不是纯手写）

#### 3. 路径关系（演化逻辑验证）
- [ ] 能识别路径 B 是从路径 A 的"分叉"（Mamba 挑战 Transformer）
- [ ] 能识别路径 B→A 的"融合"（Jamba = Transformer + Mamba）
- [ ] 能识别路径 C 是独立的"侧翼尝试"

### MVP 不做的事
- 不做 2500 篇 Scaling
- 不做趋势预测（先验证描述性分析，再加预测）
- 不做 LLM 批量标注（10 篇手工标注）
- 不做标注管理后台
- 不做数据库持久化

### MVP 技术路径

```
手动选定 10 篇论文 (JSON/硬编码)
  ↓
通过 OpenAlex API 获取元数据 (引用数、作者、机构、摘要)
  ↓
手工标注路径标签 (A/B/C/D)
  ↓
构建引用网络 (NetworkX，10 节点)
  ↓
生成三件事报告 (后端服务，基于元数据聚合 + LLM 总结)
  ↓
前端展示：力导向图 + 报告面板
```

### 关于"技术含量"的思考

API 获取数据确实没技术含量，但 MVP 的价值在于：
1. **路径发现逻辑**：给定一堆论文，系统能自动识别出 A/B/C/D 四条路径吗？（社区检测 + 标签 → 路径）
2. **三件事报告生成**：从原始元数据到结构化洞察的转化逻辑（聚合 + NLP + LLM）
3. **演化关系可视化**：分叉、融合、侧翼的视觉表达
4. **可扩展性验证**：10 篇的逻辑能否平滑扩展到 100 → 2500 篇

技术含量在"分析层"，不在"数据层"。

## Scaling 路线图

### 执行策略（管理层定调：先小后大）

> "先拿小的数据源试一试" —— Speaker 1
> "Scaling 过程可能有点慢周折" —— 浩林

#### Phase A：MVP 小实验（Week 1-2）
**10 篇种子论文 × 4 条路径，验证核心逻辑**

目标：
- 用上述 10 篇论文跑通完整链路
- 手工标注 4 条路径标签
- 实现三件事报告的生成逻辑（每条路径一份）
- 力导向图展示路径聚类 + 演化关系
- 验证"分析层"的逻辑是否成立

时间范围：2023-2024 模型架构层论文（以 GPT 出现为锚点）。

交付：10 篇论文的完整 Demo（图 + 报告），可演示

#### Phase B：Demo 固化（Week 3）
**200-500 篇，系统化**

目标：
- 标注管理系统（CRUD API）
- 三件事报告自动生成（后端服务）
- 前端展示报告（新 UI 组件）
- 趋势预测图表集成

交付：完整 Demo 系统，可演示

#### Phase C：批量导入（Week 4+）
**500 → 2500 篇，全量数据**

目标：
- 批量获取 + LLM 批量标注
- 性能优化（缓存/索引/异步）
- 数据持久化（数据库）
- 同步进行信号日报 + 2500 信号源导入

交付：生产级系统
