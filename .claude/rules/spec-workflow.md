# Spec 工作流规则

## Spec → Code 流程

Spec 写完后，Claude 判断是否需要对齐：
- 方案明确、没有分歧点 → 直接做
- 有多条路 / 涉及用户偏好的参数选择（如权重系数、视觉风格、交互行为）→ 先问用户一句再动手

不需要写正式 plan.md 文件，对话里确认即可。问的粒度是"我打算 X，行吗"，不是长篇方案。

### Spec 粒度标准

检验法："换一个技术栈实现这个 Spec，Spec 是否仍然有效？"
- Spec 只写 WHAT（问题、指标、约束、验收标准）
- HOW（公式、参数、架构选型、实现步骤）放 plan.md
- 如果 Spec 里出现了具体函数名/算法参数/代码结构 → 说明混入了 HOW，要拆出去

### Spec 迭代纪律

第一版 Spec 允许粗糙（只写 Problem + Success Metrics + Non-Goals）。
Claude 基于它出 Plan → Plan 暴露 Spec 盲点 → 改 Spec。跑 2-3 轮再动手。

## Spec 同步规则（SDD L2: Spec-Anchored）

| 变更类型 | 写新 spec？ | 更新旧 spec？ | 记 log？ |
|---------|:-:|:-:|:-:|
| 新视图 / 新数据模型 | Yes | — | Yes |
| 改已有视图的交互/字段/逻辑 | No | Yes（增量修改，不重写） | Yes |
| 纯样式/文案微调 | No | No | 一笔带过 |
| 补论文数据 | No | No | No |

核心原则：**Spec 和代码永远同步**。代码改了 → 回去改 spec。发现 Spec-Code 偏差 → 在对应 log 中记录。

## Spec 生命周期与目录

- 需求来源：会议反馈 → 新建 spec
- 未收敛的需求放 `specs/还没收敛/`
- 小需求放 `specs/小需求task/`
- 做完了 → 移到 `specs/归档/`

## 开发 SOP

**新功能（新视图/新数据模型）：**
1. Spec → Plan（Claude 出，用户过一眼）→ Code
2. 视觉类 → 先做 HTML 原型到 `prototypes/`，截图确认
3. 更新 STATUS.md + Log

**增量改动（改已有功能）：**
1. 直接改代码
2. 回去更新对应 spec（增量修改）
3. Log 里记决策/踩坑（如有）

**补数据（论文/taxonomy）：**
1. 从 knowledge/ 提取 → 列出结果让用户确认
2. 写入后端
3. 更新 STATUS.md

## Log 规则

### 什么时候必须写 log

- 选了方案 A 放弃了方案 B → 记**为什么**（下次会话不用重新讨论）
- 踩坑后解决了 → 记问题、原因、方案（下次不重蹈）
- 用户和 Claude 讨论后达成结论 → 如果结论改变了 Spec 或 Plan 的前提，写 log
- Plan 中途换方案 → 记为什么换

### 什么不用写 log

- 顺利完成、没意外 → 不写
- 纯样式微调 → 不写
- 讨论了但没结论（还在发散）→ 不写

### Log 格式

- 技术决策（选了什么、放弃了什么、为什么）
- 踩坑记录（问题 → 原因 → 解决方案）
- 知识发现（可复用的经验）
- Spec-Code 偏差

### Spec vs Plan vs Log 生命周期

| 文档 | 改动频率 | 改动意味着 | 丢了的后果 |
|------|---------|-----------|-----------|
| Spec | 少（需求变才改） | 需求/边界变了 | 做错东西 |
| Plan | 中（实现过程可调） | 方案变了，需求没变 | 重新想方案（成本中） |
| Log | 只增不改 | — | 重复讨论、重复踩坑 |
| STATUS | 每次会话更新 | 进度变了 | 不知道做到哪了 |
