<h3>optimize_anything：用单一 LLM 搜索框架统一六类优化问题，ARC-AGI 准确率从 32.5% 升至 89.5%</h3>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19633">optimize_anything: A General Framework for LLM-Based Optimization</a></li>
  <li><b>信号源</b>：<b>MIT Omar Khattab</b> · Microsoft Lakshya A Agrawal（一作）· Berkeley Matei Zaharia（资深合作）</li>
</ul>
<callout emoji="🎯">
  <p><b>Our Insights</b></p>
  <p>过去每个领域的优化问题都依赖专用算法——ARC-AGI 有 ARC-AGI 的搜索策略，CUDA kernel 生成有 kernel tuning 工具，云调度有调度器。optimize_anything 打破了这一假设：只要能给"文本产物"打分，LLM 就能通过迭代改写来求解，且多任务跨问题迁移（cross-task transfer）让每个任务额外受益——相关任务越多，单任务效果越好。</p>
  <p>这件事的格局意义在于两条线：(1) <b>领域专用优化软件</b>（自动调参、神经网络架构搜索、编译器 autotuning 创业公司）的差异化卖点被压缩——如果一个通用 LLM 搜索系统就能在 CUDA kernel 生成上 87% 超越 PyTorch 且打败 AlphaEvolve，那些依赖"专家经验编码进搜索策略"的工具的护城河需要重估；(2) <b>Databricks / Together AI</b> 等把 DSPy 生态（Omar Khattab 的核心工作）作为差异化的平台，此工作进一步强化了"prompt + 评分函数"作为通用编程范式的可信度，对其平台粘性是直接利好。</p>
  <p>cross-task transfer 收益随相关任务数量扩大这一结论，意味着拥有更多任务数据的平台比单任务工具有结构性优势——这是"数据飞轮"在优化层的一个新表现形式。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>约谈在做 AutoML / NAS / 编译器 autotuning 的创业团队，重新审视其与通用 LLM 优化框架的差异化边界是否仍然成立。</li>
    <li>跟进 Databricks GEPA 项目进展，评估 optimize_anything 开源后对 DSPy 生态平台商业化节奏的催化效果。</li>
    <li>关注 cross-task transfer 在工业场景（如多租户 SaaS 平台）中的实际收益复现，若可验证则更新对"任务数据积累"作为壁垒的判断。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>Omar Khattab 等提出 optimize_anything，将各类优化问题统一抽象为"给定文本产物（artifact）和评分函数，由 LLM 驱动迭代改进"的形式，覆盖 ARC-AGI 抽象推理基准、CUDA kernel 生成、云调度、数学组合优化等六类跨域任务，均达到 SOTA，并超越 AlphaEvolve 报告的圆形填充（n=26）方案。核心发现是多任务跨问题迁移能在相同预算下显著优于独立优化，且收益随相关任务数量增加而扩大。项目作为 GEPA 的一部分开源，支持多种后端。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：optimize_anything 将优化问题统一表述为"给定文本 artifact + 评分函数，LLM 驱动搜索迭代改进"，支持单任务、跨问题多任务（cross-problem transfer）和对未见输入泛化三种模式，作为 GEPA 项目开源。</li>
  <li><b>技术细节</b>：每轮迭代中 LLM 接收当前 artifact、评分反馈及可选的 actionable side information（如失败样例分析）生成改进版本；多任务模式下不同任务成功经验相互迁移；消融实验（ablation）证明带 actionable side information 的反馈比仅有分数收敛更快且最终得分更高。</li>
  <li><b>主结果</b>：单一 LLM 优化框架将 Gemini Flash 在 ARC-AGI 上准确率从 <b>32.5% 提升至 89.5%</b>，近乎三倍；生成的 CUDA kernel 中 <b>87%</b> 性能匹配或超越 PyTorch，并超越 AlphaEvolve 报告的 circle packing（n=26）方案。</li>
  <li><b>关键结果点</b>：多任务搜索通过 cross-task transfer 在相同预算下显著优于独立优化，收益随相关任务数量增加而扩大；在六类跨域任务上均达到 SOTA，覆盖 agent 架构搜索、云调度、CUDA kernel 生成及数学组合优化。</li>
</ul>

<h3>CODA：将 Transformer 中的"小算子"融入矩阵乘法尾部，消除显存往返开销</h3>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19269">CODA: Fusing Memory-Bound Operators as GEMM Epilogues</a></li>
  <li><b>信号源</b>：<b>Princeton Tri Dao</b> · Princeton Han Guo（一作）· Princeton Jack Zhang（合作）</li>
</ul>
<callout emoji="⚙️">
  <p><b>Our Insights</b></p>
  <p>Transformer 训练的时间瓶颈长期被认为是注意力计算（attention），FlashAttention 系列已解决了这一块。CODA 指出另一半被忽视的开销：LayerNorm、SwiGLU、残差叠加等"小算子"反复将中间结果写回全局显存再读回来，凭借"GEMM 输出 tile 留在片上时顺手完成后续计算"这一思路，CODA 把这些往返全部消除。</p>
  <p>格局影响落在两条线：(1) 这是 Tri Dao 继 FlashAttention 后系统性解决 Transformer 训练另一类显存带宽瓶颈的工作，若被主流框架（PyTorch / JAX）采纳，将进一步压缩训练成本——<b>训练基础设施创业公司</b>（依赖手写 kernel 作为差异化的 MLSys 工具链）的技术壁垒需要重估；(2) CODA 的接口设计刻意"窄"，使 LLM 自动生成 kernel 成为可行路径，这意味着未来"AI 写训练 kernel"的可信度进一步提升，对依赖人工 kernel 工程师的团队是隐性压力。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>跟踪 CODA 被 PyTorch / Triton 上游采纳的进度，若进入主干则评估对现有 custom kernel 服务商（如 Fireworks、Together AI kernel 团队）差异化的冲击。</li>
    <li>关注 Tri Dao 团队后续是否将 CODA 整合进 FlashAttention-3 或独立发布工具包，作为 Princeton Systems 生态投资标的的参考信号。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>Tri Dao 等提出 CODA，将 Transformer block 中 normalization、activation、residual 等受显存带宽限制（memory-bound）的算子统一重写为矩阵乘法（GEMM）epilogue 程序：GEMM 主循环完成矩阵乘后，输出 tile 仍留在片上寄存器或共享内存时，由一套可组合的 epilogue 原语（缩放、归约、逐元素变换、残差叠加）继续完成后续计算，从而消除中间张量写回全局显存再读回的往返开销，覆盖标准 Transformer 前向和反向中几乎全部非注意力计算。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：CODA 将 GPU kernel 抽象为"GEMM 主循环 + epilogue 程序"两阶段结构；框架层将原本独立算子（LayerNorm、GELU、bias add 等）代数等价地重参数化，映射为 epilogue 原语序列，消除中间 tensor 到全局显存的往返。</li>
  <li><b>技术细节</b>：epilogue 原语集合包含 scaling（逐元素缩放）、row/column reductions、pairwise transformations（逐元素双目运算）和 accumulation（残差叠加），足以表达 Transformer 前向中 LayerNorm、SwiGLU、dropout、residual add 及反向对应梯度算子；接口约束刻意保持"窄"，使自动调度和 LLM 代码生成都能在受限搜索空间内找到高效实现。</li>
  <li><b>主结果</b>：将 Transformer block 中 memory-bound 算子统一表达为 GEMM-epilogue 原语，tile 留在片上完成计算，减少全局显存往返，覆盖前向/反向几乎全部非注意力计算。</li>
  <li><b>关键结果点</b>：人工编写与 LLM 自动生成的 CODA kernel 在主流 Transformer workload 上均取得高性能，验证了框架级易用性与硬件级效率可同时兼得。</li>
</ul>

<h3>PEEK：为 LLM Agent 构建"导航地图"缓存，长文档推理提升 34%、成本降低 5.8x</h3>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19932">PEEK: Context Maps for Long-Context LLM Agents</a></li>
  <li><b>信号源</b>：<b>MIT Omar Khattab</b> · ETH Zhuohan Gu（一作）· MIT Samuel Madden（资深合作）</li>
</ul>
<callout emoji="🧠">
  <p><b>Our Insights</b></p>
  <p>几乎所有 long-context agent 的效率优化都在"压缩输入"或"加速注意力"上做文章，默认缓存的对象是原始文档片段或过去轨迹。PEEK 的认知反转是：agent 反复操作同一外部上下文时，真正值得缓存的不是内容本身，而是"如何导航该上下文的结构性知识"——哪些实体频繁出现、哪些 schema 被反复引用、哪些路径总是有效。</p>
  <p>这对 agentic 应用赛道的影响是具体的：(1) 依赖长上下文 + 高迭代次数的代码 agent（如 Codex 生产级编程 agent）和文档分析 agent，其运营成本被 PEEK 的机制直接压缩 <b>1.7–5.8x</b>——这是对"长上下文即护城河"论点的挑战，因为减少迭代次数的价值比单纯扩展上下文窗口更大；(2) <b>Databricks / Together AI</b> 等做 DSPy / agent 框架的平台可以将 context map 作为原生优化层纳入，而非留给用户自行实现，这是平台差异化的一个具体锚点。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>评估已投/接触的代码 agent 或文档处理 agent 标的，询问其当前 long-context 成本结构，判断 PEEK 机制是否直接影响其单位经济模型。</li>
    <li>跟进 Omar Khattab 团队是否将 PEEK 集成进 DSPy 框架；若合并，则作为 Databricks 平台竞争力强化的正向信号。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>PEEK 为反复操作同一外部上下文的 LLM agent 构建一个固定 token 预算的 context map（上下文地图）作为 prompt 中的持久化组件，缓存"方向知识"（orientation knowledge）而非原始内容——即如何导航该上下文的结构性知识。在长文档推理和代码仓库任务上，PEEK 相比强 baseline 提升 6.3–34.0%，减少 93–145 次迭代，与最强 prompt-learning 框架 ACE 相比成本降低 1.7–5.8x；在 context learning 任务上解题率提升 6.0–14.0%，成本降低 1.4x，并在多种 LM 和 agent 架构（包括 OpenAI Codex 生产级编程 agent）上均可泛化。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：PEEK 维护固定 token 预算的 context map，由三个模块组成：Distiller 从推理时信号提取可复用的方向知识；Cartographer 将其转化为结构化地图编辑；Evictor 通过优先级机制管理知识的保留与淘汰，确保不超过预算。</li>
  <li><b>技术细节</b>：Context map 是常量大小的 artifact，随每次 agent 调用不断更新；Distiller 观察推理轨迹提取"哪些实体、常量、schema 被频繁使用"等信号；Cartographer 将信号转为对 map 的结构化 patch；Evictor 类似 LRU cache 按优先级淘汰低价值条目。</li>
  <li><b>主结果</b>：PEEK 在 long-context 推理和信息聚合任务上相比强 baseline 提升 <b>6.3–34.0%</b>，同时减少 <b>93–145 次</b>迭代；与 ACE 相比成本降低 <b>1.7–5.8x</b>。</li>
  <li><b>关键结果点</b>：在 context learning 任务上，解题率提升 <b>6.0–14.0%</b>，rubric 准确率提升 <b>7.8–12.1%</b>，成本降低 <b>1.4x</b>；性能和效率收益在多种 LM 和 agent 架构上均可泛化。</li>
</ul>

<h4>optimize_anything：LLM 驱动的通用文本优化框架，ARC-AGI 准确率从 32.5% 升至 89.5%</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19633">optimize_anything: A General Framework for LLM-Based Optimization</a></li>
  <li><b>信号源</b>：<b>MIT Omar Khattab</b> · Microsoft Lakshya A Agrawal（一作）· Berkeley Matei Zaharia（资深合作）</li>
</ul>
<callout emoji="🎯">
  <p><b>Our Insights</b></p>
  <p>过去每个领域的优化问题都依赖专用算法。optimize_anything 打破这一假设：只要能给"文本产物"打分，LLM 就能迭代改写求解，且 cross-task transfer（跨任务迁移）使相关任务越多单任务效果越好——这改变了"优化工具需要领域专业化"的基本前提。</p>
  <p><b>领域专用优化软件</b>（自动调参、神经网络架构搜索、编译器 autotuning 创业公司）的差异化卖点被压缩：CUDA kernel 生成上 87% 超越 PyTorch 且打败 AlphaEvolve，说明"专家经验编码进搜索策略"的工具护城河需要重估。<b>Databricks / Together AI</b> 等 DSPy 生态平台则从中受益，"prompt + 评分函数"作为通用编程范式的可信度进一步强化，平台粘性是直接利好。cross-task transfer 收益随任务数量扩大意味着拥有更多任务数据的平台比单任务工具有结构性优势——这是"数据飞轮"在优化层的新表现。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>约谈在做 AutoML / 神经网络架构搜索（NAS）/ 编译器 autotuning 的创业团队，重新审视其与通用 LLM 优化框架的差异化边界。</li>
    <li>跟进 Databricks GEPA 项目进展，评估 optimize_anything 开源后对 DSPy 生态平台商业化节奏的催化效果。</li>
    <li>关注 cross-task transfer 在工业场景中的实际收益复现，若可验证则更新对"任务数据积累"作为壁垒的判断。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>Omar Khattab 等提出 optimize_anything，将各类优化问题统一抽象为"给定文本 artifact 和评分函数，由 LLM 驱动搜索迭代改进"的形式，覆盖 ARC-AGI 抽象推理基准、CUDA kernel 生成、云调度、数学组合优化等六类跨域任务，均达到 SOTA，并超越 AlphaEvolve 报告的圆形填充（n=26）方案。核心发现是多任务跨问题迁移在相同预算下显著优于独立优化，且收益随相关任务数量增加而扩大。项目作为 GEPA 的一部分开源。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：optimize_anything 将优化问题统一表述为"给定文本 artifact + 评分函数，LLM 驱动搜索迭代改进"，支持单任务、cross-problem transfer（跨问题迁移）和对未见输入泛化三种模式，作为 GEPA 项目开源，支持多种后端。</li>
  <li><b>技术细节</b>：每轮迭代中 LLM 接收当前 artifact、评分反馈及可选的 actionable side information（如失败样例分析）生成改进版本；多任务模式下不同任务成功经验相互迁移；消融实验证明带 actionable side information 的反馈比仅有分数收敛更快且最终得分更高。</li>
  <li><b>主结果</b>：单一 LLM 优化框架将 Gemini Flash 在 ARC-AGI 上准确率从 <b>32.5% 提升至 89.5%</b>，近乎三倍；生成的 CUDA kernel 中 <b>87%</b> 性能匹配或超越 PyTorch，并超越 AlphaEvolve 的 circle packing（n=26）方案。</li>
  <li><b>关键结果点</b>：多任务搜索通过 cross-task transfer 在相同预算下显著优于独立优化，收益随相关任务数量增加而扩大；在六类跨域任务上均达到 SOTA，覆盖 agent 架构搜索、云调度、CUDA kernel 生成及数学组合优化。</li>
</ul>

<h4>PEEK：为 Long-Context LLM Agent 构建"导航地图"缓存，提升 34%、成本降低 5.8x</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19932">PEEK: Context Maps for Long-Context LLM Agents</a></li>
  <li><b>信号源</b>：<b>MIT Omar Khattab</b> · ETH Zhuohan Gu（一作）· MIT Samuel Madden（资深合作）</li>
</ul>
<callout emoji="🧠">
  <p><b>Our Insights</b></p>
  <p>几乎所有 long-context agent 的效率优化都在"压缩输入"或"加速注意力"上做文章。PEEK 的认知反转是：agent 反复操作同一外部上下文时，真正值得缓存的不是内容本身，而是"如何导航该上下文的结构性知识"——这让 context window（上下文窗口）本身的大小变得不那么重要。</p>
  <p>对 agentic 应用赛道的影响具体在于：(1) 依赖长上下文 + 高迭代次数的代码 agent 和文档分析 agent，运营成本被直接压缩 <b>1.7–5.8x</b>——"长上下文即护城河"的论点被削弱，因为减少迭代次数的价值比单纯扩展上下文窗口更大；(2) <b>Databricks / Together AI</b> 等做 DSPy / agent 框架的平台可以将 context map 作为原生优化层纳入，而非留给用户自行实现，这是平台差异化的具体锚点。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>评估已投/接触的代码 agent 或文档处理 agent 标的，询问其当前 long-context 成本结构，判断 PEEK 机制是否直接影响其单位经济模型。</li>
    <li>跟进 Omar Khattab 团队是否将 PEEK 集成进 DSPy 框架；若合并，则作为 Databricks 平台竞争力强化的正向信号。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>PEEK 为反复操作同一外部上下文的 LLM agent 构建固定 token 预算的 context map（上下文地图）作为 prompt 持久化组件，缓存"如何导航该上下文的结构性知识"而非原始内容。在长文档推理和代码仓库任务上相比强 baseline 提升 6.3–34.0%，减少 93–145 次迭代；与最强 prompt-learning 框架 ACE 相比成本降低 1.7–5.8x；在 context learning 任务上解题率提升 6.0–14.0%，成本降低 1.4x，并在包括 OpenAI Codex 生产级编程 agent 在内的多种架构上均可泛化。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：PEEK 维护固定 token 预算的 context map，由 Distiller（从推理时信号提取可复用方向知识）、Cartographer（转化为结构化地图编辑）、Evictor（通过优先级机制管理知识保留与淘汰）三个模块组成。</li>
  <li><b>技术细节</b>：Context map 是常量大小的 artifact，随每次 agent 调用不断更新；Distiller 提取"哪些实体、常量、schema 被频繁使用"等信号；Cartographer 将信号转为结构化 patch；Evictor 类似 LRU cache 按优先级淘汰低价值条目，确保始终不超过预算。</li>
  <li><b>主结果</b>：PEEK 在 long-context 推理和信息聚合任务上相比强 baseline 提升 <b>6.3–34.0%</b>，同时减少 <b>93–145 次</b>迭代；与 ACE 相比成本降低 <b>1.7–5.8x</b>。</li>
  <li><b>关键结果点</b>：在 context learning 任务上，解题率提升 <b>6.0–14.0%</b>，rubric 准确率提升 <b>7.8–12.1%</b>，成本降低 <b>1.4x</b>；性能收益在多种 LM 和 agent 架构上均可泛化。</li>
</ul>

<h4>代码预训练不等于推理增强：结构化推理轨迹才是数学能力提升的真正来源</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19762">Rethinking Code Pretraining for Math Reasoning</a></li>
  <li><b>信号源</b>：<b>OSU Kai Zhang</b> · ETH Yuze Zhao（一作）· ETH Enhong Chen（资深合作）</li>
</ul>
<callout emoji="🧠">
  <p><b>Our Insights</b></p>
  <p>"加入代码训练数据能提升推理能力"几乎已成为预训练数据配方的默认共识。这篇论文用 10T token 受控实验拆穿了这个假设：真正有效的不是可执行代码本身，而是代码与自然语言交织的结构化推理轨迹（如 code-text / math-text 混合）。纯可执行代码提升编程，但与复杂数学推理存在竞争关系。</p>
  <p>这对预训练数据配方服务商和基础模型公司均有影响：(1) 如果"买代码数据"的逻辑不再成立，那些以代码数据资产为核心壁垒的数据供应商需要重新定义产品价值——<b>结构化推理轨迹数据</b>（含步骤标注的 math-text trace）的稀缺性和价值被本文显著抬升；(2) 专家混合（MoE）架构的路由分析首次提供了"数据域竞争"的机制层证据——这对解释为什么某些数据配方有效、某些无效提供了可观测抓手，对 MoE 架构基础模型的数据优化有直接参考价值。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>联系专注结构化推理数据生成/标注的团队（如 math trace 合成数据公司），评估其数据资产的稀缺性是否因本文结论而上升。</li>
    <li>在评估基础模型公司预训练数据配方时，主动询问其 code-text vs. math-text 混合比例及结构化样本密度，作为数学推理能力可信度的验证维度。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>该研究在 10T token 预训练语料上进行受控实验，系统拆解"代码训练提升推理"这一广泛假设，发现纯可执行代码显著提升编程能力但不能作为通用推理增强器，且与复杂数学推理存在竞争关系；真正有效的是代码与自然语言交织的结构化推理轨迹（code-text / math-text 混合）。在固定数学 budget 内提高结构化数学样本密度，可大幅提升困难数学推理同时基本保持编程性能；MoE routing（专家混合路由）分析提供了数据域竞争与协同交互的机制层证据。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：在 10T-token 预训练语料上进行受控实验，对 code、math、text 域进行细粒度分离，系统改变数据构成（纯代码、code-text 混合、math-text 混合、结构化数学密度等），引入 MoE expert activation routing 分析揭示机制。</li>
  <li><b>技术细节</b>：严格隔离 Code-NL（代码与自然语言混合）数据和纯可执行代码数据，避免过去实验中的混淆因素；在 math budget 固定前提下调节结构化数学样本（含推理步骤的 math-text trace）密度；利用 MoE 模型的 expert 激活模式观察不同 domain 数据的竞争或协同关系。</li>
  <li><b>主结果</b>：纯可执行代码显著提升编程能力，但不能作为通用推理增强器，且与复杂数学推理存在竞争关系；归因于代码的推理收益更好地被跨域结构化推理轨迹（code-text / math-text 混合）解释。</li>
  <li><b>关键结果点</b>：在固定数学 budget 内提高结构化数学样本密度，可大幅提升困难数学推理同时基本保持编程性能；MoE routing 分析提供了竞争与协同交互的机制层证据。</li>
</ul>

<h4>TDAA：首次在"既定向又下游无关"的严格威胁模型下攻击自监督预训练编码器</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19446">Targeted Downstream-Agnostic Adversarial Attacks on Pre-trained Encoders</a></li>
  <li><b>信号源</b>：<b>Microsoft Yi Zhang</b> · Peking Zhuxin Lei（一作）· ETH Ziyuan Yang（合作）</li>
</ul>
<callout emoji="🔬">
  <p><b>Our Insights</b></p>
  <p>现有下游无关对抗攻击（DAA）只需改变预测即可；TDAA 要求攻击结果指向特定目标——这把安全威胁从"随机破坏"升级为"精准伪装"，大幅提高了攻击者的实际威胁能力。研究揭示了 self-supervised pre-trained encoder（自监督预训练编码器）在更强威胁模型下的深层脆弱性。</p>
  <p>对 AI 安全赛道的意义：在 10 种自监督方法和 3 个 benchmark 数据集上的验证说明这不是单一模型漏洞，而是整个 self-supervised 预训练范式的系统性安全隐患。对于将 CLIP / DINO 等预训练编码器部署在安全敏感场景（医疗影像、内容审核）的企业来说，这是实际风险升级信号；对投资 AI 安全检测/防御工具的团队而言，这类研究是需求侧新增的正向催化剂。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>关注将 CLIP / DINO 等编码器用于安全敏感下游任务（内容审核、医疗影像）的标的，评估其对抗鲁棒性是否在产品风险矩阵中得到了充分考量。</li>
    <li>跟踪 AI 安全防御工具（adversarial robustness 方向）的融资动向，TDAA 类研究的积累通常先于这一赛道的需求爆发。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>该研究提出 TDAA（Targeted Downstream-Agnostic Adversarial Attack）框架，在不知晓下游任务的情况下对预训练编码器实施既定向又下游无关的对抗攻击，通过预先选定的"threat image"作为特征级锚点，训练一个生成器为每张输入图像生成样本专属（example-specific）扰动，使被攻击编码器对生成的对抗样本与 threat image 输出一致的特征表示。在 10 种自监督方法（包括对比学习类编码器）和 3 个 benchmark 数据集上验证有效性，定向攻击成功率显著优于现有 DAA 基线，揭示了 pre-trained encoder 面对定向威胁的严峻安全隐患。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：TDAA 在攻击者预先选定 threat image 的前提下，训练 generator 为每张输入图像生成 example-specific 对抗扰动，驱使 victim encoder 对生成样本与 threat image 输出一致特征，从而在不知晓下游任务情况下实现定向攻击。</li>
  <li><b>技术细节</b>：Generator 以原始图像为输入，输出不可见专属扰动，通过最小化对抗样本与 threat image 在 encoder 特征空间中的距离进行优化；example-specific 范式相比旧方法（为所有样本生成单一共享扰动）能应对图像多样性挑战，攻击成功率和不可见性均更有保障。</li>
  <li><b>主结果</b>：首次在既定向又下游无关的严格威胁模型下成功攻击预训练编码器；在 10 种 self-supervised 方法、3 个 benchmark 数据集上验证有效性，定向攻击成功率显著优于现有 DAA 基线。</li>
  <li><b>关键结果点</b>：揭示了 self-supervised pre-trained encoder 面对定向威胁时的显著安全隐患，为 encoder 安全防御研究提供了新的威胁模型参照。</li>
</ul>

<h3>多模态智能 (Multimodal Intelligence)</h3>

<h4>DarkLLM：用 1B 参数 LLM 将自然语言攻击指令转化为视觉对抗扰动，成功攻击 CLIP、SAM 及前沿 LLM</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.18868">DarkLLM: A Unified Adversarial Attack Framework via Language Instructions</a></li>
  <li><b>信号源</b>：<b>UCSC Xin Wang</b> · Harvard Ye Sun（一作）· Google Yu-Gang Jiang（资深合作）</li>
</ul>
<callout emoji="🔬">
  <p><b>Our Insights</b></p>
  <p>此前对多模态基础模型的对抗攻击需要专门定制每类任务的攻击策略。DarkLLM 的格局意义在于：凭借将 instruction tuning（指令微调）引入攻击生成，使攻击者只需用自然语言描述攻击意图——这让"攻击门槛"从"需要 ML 专家"降至"会写提示词即可"。</p>
  <p>在 4 个任务、13 个数据集、15 个异构模型上的验证说明这不是个别模型的漏洞，而是 CLIP / SAM 等视觉基础模型的共性弱点。对 AI 安全赛道的投研判断：(1) 多模态基础模型在内容审核、身份验证等安全敏感场景的部署风险被本文显著抬升；(2) <b>AI 对抗鲁棒性</b>防御工具的需求侧逻辑进一步强化——这类研究通常先于防御产品的融资热潮出现，是提前布局的先行指标。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>关注将 CLIP / SAM 用于内容安全审核或身份验证的企业客户，评估 DarkLLM 类攻击是否已在其安全评估体系中被考量。</li>
    <li>跟踪 adversarial robustness 防御工具创业公司的最新融资，DarkLLM 的发布是该赛道需求催化剂。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>DarkLLM 提出统一的对抗攻击框架，将 LLM 的自然语言理解能力引入对抗攻击领域，使攻击者只需用文字描述攻击意图，模型自动生成对应的视觉扰动。仅用 1B 参数的 LLM 通过 instruction tuning 统一四类攻击任务（定向攻击、无定向攻击、分割任务攻击、多模型同步攻击），在 4 个任务、13 个数据集、15 个模型上验证有效性，成功攻击 CLIP、SAM 及前沿 LLM，证明现代视觉与多模态基础模型存在可被语言驱动对抗攻击系统性利用的共性漏洞。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：DarkLLM 分为两阶段：首先通过 instruction tuning 训练 1B 参数 LLM，使其将自然语言攻击描述映射为连续潜在攻击向量；随后由视觉解码器将该向量转换为像素级对抗扰动，统一 4 类攻击任务于单一模型下。</li>
  <li><b>技术细节</b>：LLM 接收攻击者自然语言指令，输出隐向量经可学习解码头映射到图像扰动空间，扰动幅度受 L∞ 约束控制；训练时以各任务的对抗损失作为监督信号，实现类似"条件生成"的攻击生产流程。</li>
  <li><b>主结果</b>：DarkLLM 仅用 <b>1B</b> 参数 LLM 即可将自然语言攻击指令解码为视觉对抗扰动，统一 4 类攻击任务；在 4 个任务、13 个数据集、15 个模型上验证有效性，成功攻击 CLIP、SAM 及前沿 LLM。</li>
  <li><b>关键结果点</b>：自然语言 instruction tuning 使攻击具备灵活可控性，同一框架可跨异构模型生成针对性扰动；证明现代视觉与多模态基础模型存在可被语言驱动对抗攻击系统性利用的共性漏洞。</li>
</ul>

<h3>世界模型 (World Model)</h3>

<h4>SWEET：用图像编辑替代密集视频生成，稀疏关键帧规划指导机器人操控任务</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19319">SWEET: Sparse World Models via Image Editing for Embodied Manipulation</a></li>
  <li><b>信号源</b>：NUS Mike Zheng Shou（资深合作）· Google Yiren Song（一作）· MIT Yihan Wang（合作）</li>
</ul>
<callout emoji="🤖">
  <p><b>Our Insights</b></p>
  <p>具身控制的主流路径是"生成密集视频预测未来状态再提取动作"，但这条路的计算代价高昂，且大量帧信息对操控任务是冗余的。SWEET 的核心反转是：凭借将 image editing 模型（FLUX-Kontext）用于只预测少量关键视觉状态（keyframe），实验表明 image editing 在相同机器人数据设置下比 video generation（Wan2.2）产生更可靠的任务级关键帧，视觉保真度更高且推理开销显著更低。</p>
  <p>这对机器人世界模型赛道的格局影响是：(1) 依赖"视频生成模型即世界模型"假设的创业公司（如 video world model 路线的 embodied AI 团队）需要重新评估其架构选择——image editing 模型在操控任务上是更经济的替代路径；(2) <b>FLUX-Kontext</b> 类 image editing 基础模型在机器人应用场景中的价值被本文提升，持有 image editing 核心能力的模型公司在具身 AI 赛道的卡位意义需要重估。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>在接触具身 AI / 操控机器人创业公司时，主动询问其世界模型路径选择（video generation vs. image editing），判断其架构是否与本文结论一致。</li>
    <li>评估 Black Forest Labs（FLUX 系列）在机器人应用场景的战略布局，这是 image editing 模型进入具身 AI 赛道的重要信号。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>SWEET 提出稀疏视觉规划框架，用 image editing 模型（FLUX-Kontext）替代 dense video generation 作为稀疏视觉世界模型，通过语言指令和可选的箭头空间引导逐步生成 manipulation keyframe（操控关键帧）序列，再由 goal-conditioned diffusion action predictor 将相邻关键帧对转化为可执行动作块。对比实验表明 FLUX-Kontext 在相同机器人数据设置下比 Wan2.2 产生更可靠的任务级关键帧；在 DROID 和 RoboMimic 基准上验证了已见/未见场景下关键帧预测质量的提升。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：SWEET 是两阶段稀疏视觉规划框架：第一阶段用 image editing 模型（FLUX-Kontext）以 one-shot 方式在语言指令和空间箭头引导下从初始观测出发逐步生成 task-level manipulation keyframe；第二阶段用 goal-conditioned diffusion action predictor 将相邻关键帧对转化为可执行 action chunks。</li>
  <li><b>技术细节</b>：关键帧生成阶段采用 successive image editing（逐步图像编辑），每次将当前帧作为输入结合语言子目标和可选箭头空间引导生成下一关键状态；为解决编辑图像与真实图像的分布差异，训练时混合使用真实帧和经过质量过滤的编辑帧作为 subgoal 目标，即 mixed-training 策略。</li>
  <li><b>主结果</b>：对比实验表明 image editing 模型 FLUX-Kontext 相比 video generation 模型 Wan2.2 在相同机器人数据设置下产生更可靠的任务级关键帧，视觉保真度更高且推理开销显著更低。</li>
  <li><b>关键结果点</b>：SWEET 在 DROID 和 RoboMimic 基准上提升了已见和未见场景的关键帧预测质量，验证了从关键帧规划到可执行机器人动作的完整流水线的可行性。</li>
</ul>

<h3>AI infra</h3>

<h4>CODA：将 Transformer 中 memory-bound 算子重写为 GEMM-epilogue 程序，消除全局显存往返</h4>
<ul>
  <li><b>论文</b>：<a href="https://arxiv.org/abs/2605.19269">CODA: Composable Operators as GEMM Epilogues for Efficient Transformer Training</a></li>
  <li><b>信号源</b>：<b>Princeton Tri Dao</b> · Princeton Han Guo（一作）· Princeton Jack Zhang（合作）</li>
</ul>
<callout emoji="⚙️">
  <p><b>Our Insights</b></p>
  <p>Transformer 训练的时间瓶颈注意力（attention）那块已被 FlashAttention 系列解决。CODA 指出另一半被忽视的开销：LayerNorm、SwiGLU、残差叠加等"小算子"反复将中间结果写回全局显存再读回来——CODA 凭借"GEMM 输出 tile 留在片上时顺手完成后续计算"这一架构抽象，把这些往返全部消除。</p>
  <p>格局影响：(1) 这是 Tri Dao 系统性解决 Transformer 训练另一类显存带宽瓶颈的工作，若被 PyTorch / JAX 主干采纳，将进一步压缩训练成本，<b>依赖手写 kernel 作为差异化的 MLSys 工具链创业公司</b>的技术壁垒需要重估；(2) CODA 接口设计刻意"窄"，使 LLM 自动生成 kernel 成为可行路径，"AI 写训练 kernel"的可信度进一步提升，对依赖人工 kernel 工程师的团队是隐性压力。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>跟踪 CODA 被 PyTorch / Triton 上游采纳的进度，若进入主干则评估对现有 custom kernel 服务商差异化的冲击。</li>
    <li>关注 Tri Dao 团队是否将 CODA 整合进 FlashAttention-3 或独立发布工具包，作为 Princeton Systems 生态投资标的的参考信号。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>CODA 将 Transformer block 中受显存带宽限制的算子（LayerNorm、SwiGLU、dropout、residual add 等）统一重写为 GEMM-epilogue 程序：GEMM 主循环固定负责矩阵乘，epilogue 阶段在输出 tile 仍留在片上寄存器/共享内存时，执行一套可组合原语（缩放、归约、逐元素变换、残差叠加）完成后续计算，从而消除中间张量写回全局显存再读回的往返开销，覆盖标准 Transformer 前向和反向中几乎全部非注意力计算。人工编写与 LLM 自动生成的 CODA kernel 在主流 Transformer workload 上均取得高性能。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：CODA 将 GPU kernel 抽象为"GEMM 主循环 + epilogue 程序"两阶段结构；框架层将原本独立算子代数等价地重参数化，映射为 epilogue 原语序列，消除中间 tensor 到全局显存的往返。</li>
  <li><b>技术细节</b>：epilogue 原语集合包含 scaling、row/column reductions、pairwise transformations 和 accumulation，足以表达 Transformer 前向中 LayerNorm、SwiGLU、dropout、residual add 及反向对应梯度算子；接口刻意保持"窄"，使自动调度和 LLM 代码生成都能在受限搜索空间内找到高效实现。</li>
  <li><b>主结果</b>：将 Transformer block 中 memory-bound 算子统一表达为 GEMM-epilogue 原语，tile 留在片上完成计算，减少全局显存往返，覆盖前向/反向几乎全部非注意力计算。</li>
  <li><b>关键结果点</b>：人工编写与 LLM 自动生成的 CODA kernel 在主流 Transformer workload 上均取得高性能，验证了框架级易用性与硬件级效率可同时兼得。</li>
</ul>

<h4>EEG Microstates 通用表示学习：脑电信号的"字母表"——睡眠分期、情绪识别、运动想象分类全面超越传统特征</h4>
<ul>

  <li><b>信号源</b>：<b>UCSC Xin Wang</b> · MIT Xinyang Tian（一作）· Waterloo Xuesong Chen（资深合作）</li>
</ul>
<callout emoji="🧬">
  <p><b>Our Insights</b></p>
  <p>脑机接口（BCI）领域长期面临的挑战是：不同任务、不同设备的脑电信号（EEG）特征互不通用，每个应用场景都需要重新设计特征提取方案。该研究的核心 insight 是：用大规模医疗 EEG 数据训练通用 microstate tokenizer，将连续 EEG 离散化为"字母表"，让所有下游任务共用同一套表示基础——这是 NLP tokenizer 思路在神经信号处理领域的直接迁移。</p>
  <p>在睡眠分期、情绪识别、运动想象三类任务上均优于传统时域/频域特征说明 microstate 表示具有跨任务泛化性。对 BCI 和神经科技投资的意义：(1) "通用 EEG 基础模型"赛道的可行性被进一步验证，持有大规模医疗 EEG 数据的机构（大型医院、睡眠监测公司）的数据资产价值需要重估；(2) 该技术路径对做<b>消费级脑机接口</b>（Emotiv、Neurosity 类公司）和<b>临床级 BCI</b>（Synchron、BrainGate）均有潜在影响，但需关注从实验室到临床部署的工程落地差距。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>约谈专注 EEG 基础模型或神经信号处理平台的创业团队，评估其数据积累规模是否足以支撑通用 microstate tokenizer 的训练需求。</li>
    <li>关注 UCSC Xin Wang 团队后续是否开源通用 microstate tokenizer，若开源则作为 BCI 应用赛道竞争格局的重要变量跟踪。</li>
  </ul>
</callout>
<h5>摘要</h5>
<p>该研究提出基于 EEG microstates（脑电微状态）的通用表示学习方法，以大规模医疗 EEG 数据集为基础，对连续 EEG 信号进行聚类构建离散 microstate tokenizer，将多通道 EEG 在极短时间窗内的拓扑模式聚类为有限个离散状态，形成类似文本 token 的离散序列，再迁移到多种下游脑机接口（BCI）任务的表示学习。在睡眠分期、情绪识别、运动想象分类三类任务上均优于传统时域和频域特征，同时提供更好的可解释性与可扩展性。</p>
<h5>方法与结果</h5>
<ul>
  <li><b>方法框架</b>：两阶段方法——首先从大规模医疗 EEG 数据集出发，对连续 EEG 信号聚类构建离散 microstate tokenizer；随后将该 tokenizer 作为通用前端，统一应用于多种下游 BCI 任务的表示学习。</li>
  <li><b>技术细节</b>：microstate tokenizer 将多通道 EEG 在极短时间窗内的拓扑模式聚类为有限个离散状态（microstates），形成离散序列，后续模型直接在这一序列上进行序列建模；类似将连续音频波形量化为 VQ-VAE codebook 再送入语言模型处理。</li>
  <li><b>主结果</b>：microstate 表示在睡眠分期、情绪识别、运动想象分类三类任务中均优于传统时域和频域特征基线，同时提供更好的可解释性。</li>
  <li><b>关键结果点</b>：通用 microstate tokenizer 在多个模型和多类任务上均超越传统特征基线，验证了其有效性与可扩展性；具备跨任务泛化能力，适用于认知神经科学与临床研究场景。</li>
</ul>
