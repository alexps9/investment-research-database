<h3>hardmaru 提出 Diffusion 类比分块训练法，无需全网反向传播即可匹配端到端性能</h3>
<ul>
  <li><b>论文</b>：<a href="https://x.com/hardmaru/status/2059648995132367277">Block-wise Training via Diffusion Denoising Analogy（ICLR 2026）</a></li>
  <li><b>信号源</b>：<b>hardmaru</b>（个人）</li>
</ul>

<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>
  <p>深度模型训练的显存瓶颈一直被默认绑定在全局反向传播（backpropagation）上——要同时在显存中持有整个网络的中间激活才能完成梯度计算。这篇 ICLR 2026 论文把前向传播重新解读为扩散模型（diffusion model）的逐步去噪过程，每次只训练一个隔离的网络块，块与块之间不共享梯度链。<b>这意味着"训练深度模型必须持有整网激活"这个默认前提正在被动摇</b>——在 ViT、DiT 和 LLM 三类主流架构上，逐块训练的精度与端到端反向传播持平，同时峰值显存大幅下降。如果这套框架经过更大规模验证，内存受限场景下的模型训练成本结构可能需要重新评估；围绕高带宽内存（HBM）容量供给来定价训练算力的逻辑，权重也可能发生变化。</p>
</callout>

<p>　　当前大型模型训练的标准路径是端到端反向传播，这要求在前向过程中把每一层的中间激活全部保留在显存里，等梯度从最终损失函数逆向传回来。这在模型层数越来越多、批次越来越大的今天，成为单卡和多卡训练的共同瓶颈。</p>
<p>　　hardmaru 团队提出将网络的每个块单独视为一个"去噪目标"，用扩散模型的训练信号替代跨层梯度，让每个块拥有独立的局部学习信号。在 ViT、DiT 和 LLM 三类架构的实验中，这一方法与端到端训练精度持平，峰值显存占用显著降低。论文发表于 ICLR 2026。</p>

<h3>华为提出 τ 缩放定律（Tau Scaling Law），以信号传输时间常数替代晶体管密度衡量芯片进步</h3>
<ul>
  <li><b>来源</b>：<a href="https://x.com/rohanpaul_ai/status/2059645287879364854">Rohan Paul 转述华为直播内容</a></li>
  <li><b>信号源</b>：<b>Rohan Paul</b>（独立研究者，转述 Huawei 直播）</li>
</ul>

<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>
  <p>华为正在尝试争夺<b>芯片进步叙事的话语权</b>——把衡量标准从"制程节点缩了多少"换成 τ（RC 时间常数，即信号在芯片内部走完一段路径所需的时间）。配套技术 LogicFolding 通过把数字、模拟和存储电路垂直堆叠，缩短关键路径的走线长度，在标准工艺节点上实现了 55% 的密度提升。论文中"the next dollar should follow τ, not nodes"一句明示了华为期待的资本配置方向。<b>如果这套框架被业界部分接受，"先进制程是芯片进步唯一锚点"的叙事会被稀释</b>；三维集成、片上互连、芯片整机栈协同设计等此前被视为辅助路径的方向，在叙事重要性上的权重可能上升。</p>
</callout>

<p>　　过去十年，半导体进步的主流叙事几乎完全绑定在光刻节点上：从 7nm 到 5nm 到 3nm，晶体管越做越小，性能和能效跟着提升。这套逻辑在西方制裁体系下对华为形成了明显约束——无法获得台积电先进节点的华为，理论上在芯片性能上永远处于下风。</p>
<p>　　华为提出 τ 缩放定律，把焦点从晶体管尺寸转向信号传播的 RC 时间常数。LogicFolding 技术通过垂直堆叠逻辑层来缩短互连走线，在现有工艺节点上实现密度和频率的双重提升。目前该内容来自华为自身的直播和论文披露，独立第三方验证尚未出现。</p>

<p><em>* 注：上述数据和主张来自华为官方直播及论文，信源为单方披露；55% 密度提升数字尚无第三方拆解或量产验证。</em></p>

  <li><b>信号源</b>：Alibaba Cloud 官方</li>
</ul>

<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>

<h3>Alibaba Cloud 发布 Agent Infra 平台，覆盖 Agent 构建、治理、可观测性与自主运维全链路</h3>
<ul>
  <li><b>来源</b>：<a href="https://x.com/alibaba_cloud/status/2059483061843091626">Alibaba Cloud 官方</a></li>
  <li><b>信号源</b>：Alibaba Cloud 官方</li>
</ul>

<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>
  <p>Agent 系统从原型到生产部署的工程化落地一直是企业采用的主要卡点，涉及多 agent 协调、可观测性、权限管控和自动运维等环节，目前市面上的解决方案大多需要拼接多个工具。Alibaba Cloud 用一套平台覆盖这四个环节：AgentRun（构建与部署）、AgentTeams（多 agent 治理）、AgentLoop（可观测性优化）和 STAROps（自主 AI 运维）。<b>这套平台的核心赌注在于：Agent 工程化的复杂度足够高，以至于开发者愿意接受一站式方案而不是自行组合</b>——如果这个判断成立，平台层的粘性将显著高于单一工具。往上游看，这把 Alibaba Cloud 的 AI 平台定位从模型提供者进一步延伸到了基础设施运营者；往下游看，独立 Agent 框架工具的替代压力会上升。</p>
</callout>

<p>　　当前 AI Agent 开发生态处于碎片化阶段——构建用 LangChain 或自研框架，部署用 Kubernetes，监控用独立 observability 工具，运维靠人工。这种碎片化在小型项目里尚可接受，但在企业级多 agent 系统中会快速变成工程负担。</p>
<p>　　Alibaba Cloud 在 QwenConference 2026 上发布 Agent Infra 平台，将四大核心组件整合为一体：AgentRun 负责一站式构建与部署，AgentTeams 处理多 agent 间的协作与权限治理，AgentLoop 提供可观测性与持续优化，STAROps 则实现自主 AI 运维。目前该平台的具体定价和可用区信息尚未完整披露。</p>

<ul>
  <li><b>信号源</b>：TechCrunch（<a href="https://x.com/TechCrunch/status/2059613241199005977">链接</a>）</li>
</ul>
<callout emoji="💼">
