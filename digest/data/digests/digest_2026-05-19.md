<h3>NVIDIA 发布 LongLive-2.0：NVFP4 量化并行推理架构，专为长视频生成设计</h3>
<ul>
  <li><b>来源</b>：<a href="https://x.com/_akhaliq/status/2056763911462891825">AK 转发 NVIDIA</a></li>
  <li><b>信号源</b>：NVIDIA（AK 转发）</li>
</ul>
<callout emoji="⚙️">
  <p><b>Our Insights</b></p>
  <p>共识锚点：长视频生成（分钟级 / 电影级）面临的核心瓶颈是显存墙和推理时延——单帧扩散模型的计算量随视频长度线性增长，现有 GPU 显存无法承载完整序列。NVIDIA 用 NVFP4 量化（4-bit 浮点）压缩激活值显存占用，并配合并行推理调度，试图将长视频生成从"实验室 demo"推向"可服务化"。投研意涵：这类基础设施工具降低了长视频生成创业公司的硬件门槛——此前只有 Runway/Pika 等融资充裕的团队能承担 H100 集群成本；若 NVFP4 推理方案成熟，中等融资规模的视频生成 startup 也能支撑分钟级输出。NVIDIA 持续在推理侧提供专有优化，强化其 CUDA 生态 lock-in。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>跟进 LongLive-2.0 是否开源或仅限 NVIDIA 生态使用，评估 AMD/国产 GPU 生态的对标能力</li>
    <li>接触视频生成方向创业团队（特别是尝试分钟级输出的），了解其对该推理框架的采纳意愿</li>
  </ul>
</callout>
<p><b>摘要</b></p>
<p>NVIDIA 发布 LongLive-2.0，基于 NVFP4 精度量化与并行计算架构，专门解决长视频生成任务中显存占用大、推理效率低的工程瓶颈。</p>

<h4>Xin Wang：Agent 安全评估应基于执行轨迹而非最终输出</h4>
<ul>
  <li><b>来源</b>：<a href="https://x.com/xwang_lk/status/2056751738078585344">Xin Wang 推文</a> · <a href="https://x.com/xwang_lk/status/2056531396488007994">HarnessAudit 介绍</a></li>
  <li><b>信号源</b>：<b>UCSB Xin Wang</b></li>
</ul>
<callout emoji="🧠">
  <p><b>Our Insights</b></p>
  <p>当前 agent 评估体系只看"任务完成了没有"，不检查执行过程中是否越权访问数据、跨组件泄露上下文。HarnessAudit 在 210 任务/8 领域/10 种 harness 配置下测试，最高安全得分仅 0.32——意味着当前主流 agent 框架在安全性上严重不及格。投研意涵：对于投资 agentic AI 的 fund，agent safety/compliance 层是尚未被充分商业化的基础设施缺口，类比 DevOps 中的安全审计工具。企业客户（金融/医疗/政府）将 agent 部署到生产环境的前提是 harness 级别的安全可审计。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>寻找专注 agent runtime security / harness auditing 方向的早期团队</li>
    <li>评估已投 agent 平台标的是否具备轨迹级安全审计能力</li>
  </ul>
</callout>
<p><b>信号内容</b></p>
<p>Xin Wang 指出 agent 安全研究过于聚焦模型对齐，忽视了 harness 架构安全——harness 决定工具调用权限和子 agent 间消息流转。HarnessAudit 框架评测边界合规、执行保真度与系统稳定性三层，测试结果显示当前 agent 系统安全表现极差。</p>

<h4>Aaron Levie：企业 Agent 的核心挑战是上下文管理</h4>
<ul>
  <li><b>来源</b>：<a href="https://x.com/levie/status/2056574979236409521">Aaron Levie 推文</a></li>
  <li><b>信号源</b>：<b>Box Aaron Levie</b></li>
</ul>
<callout emoji="💼">
  <p><b>Our Insights</b></p>
  <p>Box CEO 的观点指向企业 agent 落地的实际瓶颈：不是模型不够聪明，而是给 agent 喂什么数据、喂多少。过多冗余信息导致 hallucination，过少则 agent 无法产出有价值结果。本质上这是 knowledge management / RAG 质量问题的另一种表述。投研意涵：企业 agent 赛道的价值链中，数据治理/上下文编排层（而非 agent 框架本身）可能是真正的付费卡点。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>关注企业知识管理/RAG 优化方向的创业公司（区别于通用 agent 框架），评估其与 Box/Notion 等平台的整合路径</li>
  </ul>
</callout>
<p><b>信号内容</b></p>
<p>Levie 认为企业 agent 策略的核心挑战是为 agent 提供恰当的上下文——数据冗余或冲突会误导 agent，数据不足则限制价值上限，本质上是一个数据策略问题。</p>

<h4>Balaji：当前 AI Agent 并非真正自主，Principal-Agent 问题重演</h4>
<ul>
  <li><b>来源</b>：<a href="https://x.com/balajis/status/2056677657715286132">Balaji 推文</a></li>
  <li><b>信号源</b>：<b>Balaji Srinivasan</b></li>
</ul>
<callout emoji="🧠">
  <p><b>Our Insights</b></p>
  <p>Balaji 引用 METR 研究指出，当前所谓"自主 agent"仍需人类持续 prompting 和验证输出质量，否则代码库将充斥无意义生成物。这一判断对 agentic AI 估值叙事构成冷水：如果 agent 的实际运行仍需大量人力监督，其自动化倍率（ROI）远低于市场预期。投研意涵：对投出高估值 agentic startup 的 fund，需要重新校准"人类监督成本"在 TCO 中的占比——目前的 demo 效果和实际生产环境存在显著落差。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>要求 portfolio 中 agent 方向标的提供真实客户场景的人工监督比例数据，验证自动化率</li>
  </ul>
</callout>
<p><b>信号内容</b></p>
<p>Balaji 认为当前 AI agent 本质上仍依赖人类持续介入，引用 METR 研究批评，指出 principal-agent 问题在人类与 AI agent 之间重演。</p>

<h4>dex：Agent 开发经验——用确定性控制流替代 Prompt 逻辑控制</h4>
<ul>
  <li><b>来源</b>：<a href="https://x.com/dexhorthy/status/2056731033475936486">dex 推文</a></li>
  <li><b>信号源</b>：<b>dex</b></li>
</ul>
<callout emoji="⚙️">
  <p><b>Our Insights</b></p>
  <p>这条工程经验与 Balaji 的观点互相印证：当前 agent 在生产环境中，能用代码确定性解决的分支逻辑不应托付给 prompt——后者不可靠且难以调试。投研意涵：优秀的 agent 产品团队应展现强工程纪律（确定性流程 + LLM 仅处理模糊判断），而非过度依赖 prompt engineering。这是评估 agent startup 技术成熟度的一个指标。</p>
  <p><b>Potential To-dos</b></p>
  <ul>
    <li>在技术 DD 时追问 agent 产品中确定性代码 vs prompt 逻辑的比例</li>
  </ul>
</callout>
<p><b>信号内容</b></p>
<p>作者建议在 AI agent 开发中，能用 if/else、函数调用等确定性控制流解决的逻辑，不要依赖 prompt 指令引导模型行为。</p>

    <li>关注 YC W26/S26 batch 中 personal AI / dynamic tool-use 方向的公司</li>
