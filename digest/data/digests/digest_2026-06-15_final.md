<h4>LLM Agent 自我进化的真相：靠"背操作步骤"而非"学抽象规则"——规则被完全破坏也不影响性能</h4>
<ul>
  <li><b>信号源</b>：<b>Rohan Paul</b>（<a href="https://x.com/rohanpaul_ai/status/2066166267950858561">X</a>，转述 arXiv 论文）</li>
</ul>
<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>
  <p text-indent="1">这项研究对"自我进化 Agent"的普遍假设构成直接挑战。<b>现有 LLM Agent 的所谓"自我学习"，本质上是模仿历史操作流水——不是在提炼经验，而是在复读答案</b>。这意味着开发者精心提炼的抽象规则（"遇到 X 情况做 Y"）对 Agent 几乎没有约束力；真正驱动 Agent 行为的是原始的操作历史记录。对 Agent 系统设计者而言，这个发现的含义很直接：与其花时间写高质量规则文档，不如确保历史轨迹数据的质量与完整性。它也为 Agent memory 架构设计提出了新问题：如何让 Agent 真正"消化"规则，而不只是"储存"它？</p>
</callout>
<p text-indent="1">研究通过"秘密替换实验"来判断 LLM Agent 究竟在依赖什么记忆。实验分两组：一组悄悄把 <b>episodic memory</b>（逐步操作历史记录）替换为随机垃圾文本，另一组把 <b>semantic memory</b>（凝练的抽象规则摘要）完全破坏——观察两种情况下 Agent 的任务完成率变化。</p>
<p text-indent="1">结果显示：破坏操作历史记录导致 Agent 性能<b>显著下降</b>；而完全破坏抽象规则后，Agent 表现<b>零下降</b>。这说明 Agent 在执行任务时几乎完全忽略了规则层，核心依赖是逐步历史记录的模式复制——类似学生考试靠背例题而不是理解解题原理。</p>
<p text-indent="1">这对"Agent Lifespan Engineering"（Agent 全生命周期工程化管理）有直接含义：如果规则层不起作用，那么依赖规则来纠正 Agent 行为偏差的干预策略将收效甚微，需要从历史轨迹数据的写入和管理入手。</p>

<p>{anchor:card_2}</p>
<h4>AgingBench：长期部署的 AI Agent 会"老化"——4 类记忆衰减路径导致可靠性悄然下降</h4>
<ul>
  <li><b>信号源</b>：<b>Rohan Paul</b>（<a href="https://x.com/rohanpaul_ai/status/2066181618579214775">X</a>，转述 University of Texas 论文）</li>
</ul>
<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>
  <p text-indent="1">当前 Agent 评测几乎全部在"干净"初始状态下进行，但真实部署的 Agent 会随时间累积记忆、压缩历史、更新事实——这些操作本身就会引入错误。<b>AgingBench 填补了一个重要空白：首次系统量化 Agent 在多轮对话 / 多任务 session 之后的可靠性衰减，且发现不同衰减来源需要完全不同的修复策略</b>。这对企业级 Agent 部署尤为关键——一个在上线时表现优秀的 Agent，三个月后可能已经在悄悄犯错，而用户和开发者都察觉不到。"存储更多历史"并不是万能解法，问题可能出在压缩逻辑或检索策略上。</p>
</callout>
<p text-indent="1"><b>德克萨斯大学</b>研究团队提出 <b>AgingBench</b>，专门评估 Agent 在跨多个 session 长期运行后的可靠性衰减问题，将 Agent 的生命周期类比为"老化的操作系统"——底层 CPU（基础模型）不变，但文件系统里积累的垃圾数据越来越多。</p>
<p text-indent="1">研究识别出 4 类老化机制：<b>①</b> summarization loss（摘要压缩时丢失关键细节）；<b>②</b> memory confusion（相似记忆相互干扰）；<b>③</b> stale facts（事实更新滞后，旧知识未同步）；<b>④</b> maintenance-induced failure（维护操作意外清除有效记忆）。实验表明，Agent 可能表现上仍然"能干"，但输出精确度已悄然下滑，且不同衰减来源的根因各异，无法用同一策略覆盖。</p>

<h4>OpenRouter Fusion API 拼接三款模型逼近 Fable 5 水准——"多模型路由"成为旗舰模型下线后的应急策略</h4>
<ul>
  <li><b>信号源</b>：<b>Carlos E. Perez</b>（<a href="https://x.com/IntuitMachine/status/2066112340458619254">X</a>）</li>
</ul>
<callout emoji="💡" background-color="light-orange" border-color="orange">
  <p><b>Insights</b></p>
  <p text-indent="1"><b>这是一个关于"模型替代性"的有趣现实案例：当单一旗舰模型下线，社区通过组合多个中等模型的方式来复现其能力</b>——Gemini 3 Flash + Kimi K2.6 + DeepSeek V4 通过 OpenRouter 的 Fusion API 融合后，据称性能接近已停服的 Fable 5。这既验证了 model routing（模型路由）层的实际价值，也说明前沿能力的"组合可达性"在提升——单一提供商的断供风险正在被多源拼接策略对冲。对 infra 层玩家（如 OpenRouter）来说，这是一个扩大平台价值的窗口。</p>
</callout>
<p text-indent="1">在 <b>Fable 5</b> 停止服务后，社区用户在 <b>OpenRouter</b> 平台上尝试通过 Fusion API 将 <b>Gemini 3 Flash</b>、<b>Kimi K2.6</b> 和 <b>DeepSeek V4</b> 三款模型组合使用，据 Carlos E. Perez 反馈，拼接后的性能水平已接近 Fable 5 的使用体验。</p>
<p text-indent="1">这一现象印证了 model routing 层的核心价值：在同一任务流中智能调度不同模型，既能优化成本，也能在单一供应商断供时快速切换替代方案。OpenRouter 的 Fusion API 将这一能力产品化，降低了多模型编排的技术门槛。</p>
<p text-indent="1"><em>* 注：此为社区用户的非正式体验反馈，非官方评测数据，具体性能对比未经独立验证。</em></p>
